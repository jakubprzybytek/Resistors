# Implementation Plan — Multiple Ranked Results ("Show 3 more")

## Summary
Replace the single best/closest result with a full result set: every unique resistor combination (up to 5 resistors) whose deviation from the target is within ±5%. Results are displayed below the `Calculate` button, three at a time, with a `Show 3 more` button to reveal more, and a toggle to switch between two rankings (accuracy, resistor count) that reorders the already-computed set and resets pagination. Each calculation logs the result count and computation time to the developer console. When no results are found, an explanatory empty-state message is shown instead of the list.

## Target architecture
- **`src/network.ts`**: add a new, independent function `findAllResistorNetworks(resistors, target, opts)` that performs the same layered series/parallel BFS as the existing `findResistorNetworkUnlimited`, but:
  - collects **every** node (at any resistor count 1–5) whose value is within ±5% of target, instead of stopping at the first match.
  - treats two nodes as the **same result** only if they represent the same resistor combination (same left/right structure), not merely the same achieved value — so `100Ω + 200Ω` and a single `300Ω` leaf are both kept if both are within tolerance.
  - `findResistorNetworkUnlimited` (used elsewhere) is left **untouched** to avoid regressing existing behavior/tests; the new function duplicates the small amount of BFS scaffolding it needs rather than being layered on top of the existing one (their semantics differ: the old one needs an outside-tolerance "closest" fallback that the new all-results function doesn't track).
- **`src/resultRanking.ts`** (new pure module): `RankingMode = "accuracy" | "resistorCount"` and `sortResults(results, mode)` implementing the tie-break rules from the story.
- **`App.tsx`**: owns `results`, `ranking`, `visibleCount` (pagination cursor) state; runs the search once per `Calculate` click, logs count/time, and resets ranking-independent state (`results`, `visibleCount`) on every new calculation.
- **`ResultsList.tsx`** (new component): renders the ranking toggle, the empty-state message, the visible slice of results (each via `ResultCard`), and the `Show 3 more` button. Replaces the direct `ResultCard` usage in `App.tsx`.
- **`ResultCard.tsx`**: simplified to always render a single non-null result row (the "no result" case moves up to `ResultsList`).

## Phase 1 — Search algorithm (`network.ts`)
1. Add `export interface NetworkResult { node: Node; count: number; value: number; absError: number; deviationPct: number; description: string }`.
2. Add `export function findAllResistorNetworks(resistors: number[], target: number, opts?: { relTolerance?: number; maxResistors?: number }): NetworkResult[]`:
   - Defaults: `relTolerance = 0.05` (±5%), `maxResistors = 5`.
   - Reuse the same distinct-value extraction, `describe()`, and series/parallel `combine()` logic as `findResistorNetworkUnlimited`, building `layers[1..maxResistors]`.
   - **Every** node constructed (at every layer) is checked against the tolerance band and, if within it, pushed into the results array — regardless of whether that node continues to be used to build further layers.
   - **Growth-bucket cap (performance safeguard)**: to keep layer sizes tractable for large predefined series (e.g. E24, ~193 values) within 5 resistors, retain at most `N` (e.g. 4) representative nodes per achieved-value bucket for the purpose of building subsequent layers (today's algorithm retains exactly 1). This is a deliberate, documented trade-off — see "Resolved decisions" #6.
   - Final safety dedup: drop any result whose `description` string is an exact duplicate of one already collected (should not normally trigger, given the BFS never revisits the same `(kind, left, right)` pair twice, but guards against edge cases).
   - Returns the unsorted list; sorting is the caller's responsibility via `resultRanking.ts`.
3. `findResistorNetworkUnlimited` stays as-is; no shared code changes that could alter its behavior.

## Phase 2 — Ranking (`resultRanking.ts`)
1. `export type RankingMode = "accuracy" | "resistorCount"`.
2. `export function sortResults(results: NetworkResult[], mode: RankingMode): NetworkResult[]`:
   - `"accuracy"`: sort by `absError` ascending, tie-break by `count` ascending.
   - `"resistorCount"`: sort by `count` ascending, tie-break by `absError` ascending.
   - Remaining ties (equal on both properties): left in whatever order `Array.prototype.sort` (stable) produces — no explicit secondary key, per story.

## Phase 3 — `App.tsx` wiring
1. Replace single-result state with:
   - `results: NetworkResult[]`, `ranking: RankingMode` (default `"accuracy"`), `visibleCount: number` (default `3`), `target: number`, `hasCalculated: boolean`.
2. `handleCalculate(values, target)`:
   - Record `performance.now()` before/after calling `findAllResistorNetworks(values, target, { relTolerance: 0.05, maxResistors: 5 })`.
   - `console.log` the result count and the computation time (ms) — two separate log statements, one per story bullet, so each is independently assertable.
   - `setResults(sortResults(found, ranking))`, `setVisibleCount(3)`, `setTarget(target)`, `setHasCalculated(true)`.
3. `handleRankingChange(mode)`: `setRanking(mode)`, `setResults(prev => sortResults(prev, mode))`, `setVisibleCount(3)`.
4. `handleShowMore()`: `setVisibleCount(v => Math.min(v + 3, results.length))`.
5. Render order: `ResistorForm` first, then `{hasCalculated && <ResultsList ... />}` directly below it (below the `Calculate` button, per story).

## Phase 4 — `ResultsList` / `ResultCard` UI
1. `ResultsList` props: `results`, `visibleCount`, `ranking`, `onRankingChange`, `onShowMore`, `target`.
2. If `results.length === 0`: render an empty-state block (`role="status"`) with a message explaining that no results could be determined for the given inputs (exact copy is a minor open item — see below).
3. Otherwise:
   - Render the ranking toggle (two buttons, `role="tablist"`/`role="tab"` pattern consistent with the existing series tabs) labeled for accuracy vs. resistor count.
   - Render `results.slice(0, visibleCount).map(r => <ResultCard key={r.description} result={r} target={target} />)`, stacked vertically.
   - Render `Show 3 more` button only when `visibleCount < results.length`; clicking calls `onShowMore`.
4. `ResultCard`: change `Props.result` from `SearchResult | null` to a non-null `NetworkResult`; drop the empty-state branch (moved to `ResultsList`). Since every result is already within ±5%, replace the old `found`-driven "Exact match / Closest possible match" label with a per-result check (e.g., `absError`/`target <= 0.01` → "Exact match", else "Approximate match (within 5%)").

## Phase 5 — Styling
1. `ResultsList.scss`: `.results-list` (vertical stack, gap), `.results-list__ranking` (reuse the tab-bar look from `resistor-form__tabs`/`__tab`), `.results-list__empty` (reuse `.result-card--empty` look), `.results-list__show-more` (reuse `.resistor-form__button` look).
2. No changes needed to `ResultCard.scss` beyond what the label wording change requires (none structurally).

## Phase 6 — Testing

   - `src/__tests__/resultRanking.test.ts`: accuracy-first sort with resistor-count tie-break; resistor-count-first sort with accuracy tie-break; fully-tied entries are all preserved (set equality, order unchecked).
   - `src/components/__tests__/ResultsList.test.tsx`: renders first 3 results initially; `Show 3 more` appends the next 3 (or fewer if remaining < 3) and is hidden once exhausted; switching ranking reorders the list and resets visible count to 3; empty results render the explanatory message and no ranking toggle/list.

   - Update `tests/e2e/resistor-calculator.spec.ts` to assert the results list renders below the `Calculate` button and contains at least 3 entries for a typical input.

## Resolved decisions (from story review)
1. Tolerance for the full result set is fixed at ±5%, independent of the existing single-result `findResistorNetworkUnlimited` (which keeps its own configurable `relTolerance`, default 1%).
2. Search cap: maximum 5 resistors per combination.
3. "Unique result" = unique resistor combination (structure), not unique achieved value; same-value/different-combination results are both kept.
4. Default ranking is accuracy; ranking toggle only reorders the already-computed set and resets pagination — it never re-runs the search.
5. Console logging is two separate `console.log` calls per calculation: total result count, and computation time in milliseconds.
6. **Performance safeguard (trade-off)**: because fully exhaustive enumeration of all resistor combinations up to 5 resistors is combinatorially infeasible for large predefined series (e.g. E24 has ~193 values), the search retains a small, capped number of representative combinations per achieved-value bucket while building up from 1 to 5 resistors, rather than literally every possible combination. This keeps the feature usable in a browser at interactive speed while still surfacing multiple distinct combinations per value in the common case. This is a pragmatic reading of "no valid result should be omitted" rather than a literal guarantee for pathological inputs.
7. `ResultCard` no longer accepts a `null` result; the "no results" empty state is handled once, in `ResultsList`.

## Open questions (not blocking, but worth a quick confirmation)
1. Exact copy for the empty-state message and the ranking toggle button labels (placeholder wording proposed above; easy to change without affecting logic).
2. Whether the ranking selection should reset to "accuracy" on every new `Calculate`, or persist as a user preference across calculations — this plan assumes it **persists** (only `results`/pagination reset), since the story only mandates resetting those two.
3. Confirmation that the capped-representatives performance safeguard (#6 above) is an acceptable interpretation of "no result should be missed" for large predefined series.
