# Implementation Plan — Resistor Series (E3/E6/E12/E24 + Custom)

## Summary
Add tabbed selection of resistor value sources above the existing "available values" textarea: `E3`, `E6`, `E12`, `E24`, `Custom`. Predefined tabs show hardcoded, read-only, decade-generated values (1Ω–100MΩ) formatted with `k`/`M` suffix tokens, grouped one decade per line. The `Custom` tab keeps today's free-text behavior (comma/newline separated, now with `k`/`M` suffix parsing) and retains its text in memory across tab switches (cleared on page refresh). `Calculate` always operates on the values of the currently active tab. Any invalid token blocks calculation with an explanatory error.

## Target architecture
- **New pure logic module** `src/resistorSeries.ts`: hardcoded base values per series, decade generation (base × 10^n, n=0..8, clamped to [1, 1e8]), and formatting into decade-row text (matches predefined textarea content).
- **New pure logic module** `src/resistorValueTokens.ts`: shared value ⇄ token conversion used by both series formatting and custom parsing:
  - `parseToken(token): number | null` — parses `"4.7k"`, `"1 M"`, `"100"`, case-insensitive, optional space before suffix.
  - `formatValue(value): string` — inverse formatting (e.g. `4700 → "4.7k"`), used to render predefined rows.
  - `parseResistorValues(input): { values: number[]; invalidTokens: string[] }` — replaces the current ad-hoc `parseResistorValues` in `ResistorForm.tsx`, splitting on comma/newline, trimming, ignoring blanks, and collecting invalid tokens instead of silently producing `NaN`.
- **`ResistorForm.tsx` changes**: add tab bar state (`SeriesName | "Custom"`, default first predefined tab per story), compute textarea `value`/`readOnly` from active tab, keep a separate `customText` state so switching tabs doesn't lose custom input (in-memory only), validate using `parseResistorValues` and block submit on `invalidTokens.length > 0`.
- **Styling**: extend `ResistorForm.scss` with a `&__tabs` / `&__tab` block consistent with existing blueprint theme (border, accent color, active state), no new component file needed.

## Phase 1 — Series data & token logic (pure functions, no UI)
1. `resistorValueTokens.ts`:
   - `parseToken(token: string): number | null` — regex `^(-?\d+(\.\d+)?)\s*(k|K|m|M)?$`; `k/K` → ×1000, `m` (lowercase, per story only `k`/`M` are meaningful — treat lowercase `m` as invalid since only `M`/`m`... **decision:** suffixes are case-insensitive per story line 8, so both `k`/`K` and `m`/`M` map to kilo/mega respectively). Returns `null` for anything that doesn't match or parses to a non-positive number.
   - `formatValue(value: number): string` — value ≥ 1e6 → `${trim(value/1e6)}M`; value ≥ 1e3 → `${trim(value/1e3)}k`; else `trim(value)`. `trim` strips floating-point artifacts (`toFixed(10)` + `parseFloat`) so `1.0` renders `1`, `4.7` renders `4.7`.
   - `parseResistorValues(input: string): { values: number[]; invalidTokens: string[] }` — split on `/[,\n]+/`, trim, drop empty entries, run each through `parseToken`; non-null → `values`, null → pushed verbatim into `invalidTokens`.
2. `resistorSeries.ts`:
   - `SERIES_BASE_VALUES: Record<"E3"|"E6"|"E12"|"E24", number[]>` hardcoded per the story's Decade Values section.
   - `generateSeriesValues(series): number[]` — for `n` in `0..8`, compute `base * 10^n` for each base, filter to `[1, 1e8]`; flatten, sort ascending, dedupe (rounding to avoid float dupes).
   - `formatSeriesAsRows(series): string` — for `n` in `0..8`, build one row per decade: bases × 10^n filtered to range, mapped through `formatValue`, joined `", "`; skip empty rows; join rows with `\n`. This directly produces the story's example (`1, 2.2, 4.7` / `10, 22, 47` / `1k, 2.2k, 4.7k` / ...).

## Phase 2 — ResistorForm UI updates
1. Add `type SeriesTab = "E3" | "E6" | "E12" | "E24" | "Custom"`; `activeTab` state defaults to `"E3"` (first tab, per story).
2. Add tab bar (row of buttons/labels) above the values textarea; clicking sets `activeTab`.
3. Keep `customText` state independent of `activeTab`; textarea `value` is `activeTab === "Custom" ? customText : formatSeriesAsRows(activeTab)`; `onChange` only updates `customText` and only when `activeTab === "Custom"`; `readOnly` is `activeTab !== "Custom"`.
4. On submit, run `parseResistorValues` on the currently displayed textarea content:
   - `invalidTokens.length > 0` → error message: `` Invalid value(s): ${invalidTokens.join(", ")} `` and do not call `onCalculate`.
   - Otherwise keep existing empty/positive-number/target validation as-is.
5. Predefined tabs are generated from trusted hardcoded data, so they should never hit the invalid-token path, but reusing the same parser keeps behavior consistent and testable.

## Phase 3 — Styling
1. Add `.resistor-form__tabs` (flex row, small gap) and `.resistor-form__tab` (button styled like existing accent border, `--tab-active` state using `--bp-accent` background per existing theme variables in `App.scss`).
2. Mark active tab visually (e.g. bottom border or filled background) and disabled/read-only textarea with a subtly dimmed background to signal non-editability.

## Phase 4 — Testing
1. **Unit tests (Vitest)**:
   - `src/__tests__/resistorValueTokens.test.ts`: `parseToken` for plain numbers, `k`/`K` suffix, `M`/`m` suffix, decimals with suffix (`4.7k` → 4700), optional space (`1 k` → 1000), invalid tokens (letters, empty, multiple suffixes) → `null`; `formatValue` round-trips for representative values (1, 2.2, 100, 1000, 4700, 1_000_000, 100_000_000); `parseResistorValues` mixed valid/invalid input returns correct `invalidTokens`.
   - `src/__tests__/resistorSeries.test.ts`: for each series assert generated count (E3=25, E6=49, E12=97, E24=193 per prior analysis), min value 1, max value 1e8, ascending sorted, no duplicates; `formatSeriesAsRows("E3")` matches the exact story example (`"1, 2.2, 4.7"`, `"10, 22, 47"`, `"1k, 2.2k, 4.7k"`, ...).
   - `src/components/__tests__/ResistorForm.test.tsx`: default tab is E3 and textarea is read-only with generated content; switching to Custom shows editable empty textarea; typing in Custom then switching to E3 and back preserves the typed text; invalid token (e.g. `"100, abc"`) shows a validation error and blocks `onCalculate`.
2. **E2E tests (Playwright)**:
   - `tests/e2e/resistor-series.spec.ts`: for each of E3/E6/E12/E24, select the tab, run up to 3 calculations with different target values, assert the result card shows a value/description; assert textarea is read-only (e.g. `toHaveAttribute("readonly", ...)`) and content matches expected first row.
   - Custom tab case: comma-separated no-suffix, newline-separated, and mixed with `k`/`M` suffix inputs, each producing a valid calculation.
   - Keep existing `resistor-calculator.spec.ts` custom-flow test as regression coverage.

## Phase 5 — Validation
1. `npm test` (Vitest) passes, including new token/series/form tests.
2. `npm run test:e2e` (Playwright) passes, including new series spec.
3. Manual smoke test via `npm run dev`: switch through all 5 tabs, verify read-only predefined content and calculate behavior, verify custom text persistence across tab switches and its loss on page refresh.

## Resolved decisions (from story review)
1. Decade range: 1Ω to 100MΩ, generated as `base × 10^n` for `n = 0..8`.
2. Predefined textarea content: suffix tokens grouped one decade per line (not just grouped by suffix generally).
3. Invalid custom token behavior: blocks calculation entirely; error names the invalid value(s).
4. Custom text persistence: in-memory component state only (survives tab switching, cleared on refresh) — no `localStorage`/backend needed.
5. First tab active by default: `E3`.
6. QA scope for e2e: up to three calculations per tab with result assertions; Custom tab additionally covers separator and suffix variations.
