# Implementation Plan — Web App Initialization

## Target architecture
- **Frontend**: Vite + React + TypeScript (Vite is the natural fit for a StaticSite build and keeps builds fast).
- **Styling**: Sass (`.scss` modules per component).
- **Logic**: existing `network.ts` (resistor network search) is reused as-is, imported by a React component.
- **Infra**: SST v3 ("Ion") `sst.config.ts` using `sst.aws.StaticSite`.
- **CI/CD**: GitHub Actions workflow deploying to the `int` stage.

## Proposed file structure
```
src/
  main.tsx              // React entry point
  App.tsx                // top-level component
  App.scss
  network.ts             // moved from root, unchanged logic
  components/
    ResistorForm.tsx      // textarea + target input + button
    ResistorForm.scss
    ResultCard.tsx         // shows calculated network + deviation
    ResultCard.scss
index.html                // Vite entry HTML
sst.config.ts
.github/workflows/deploy.yml
vite.config.ts
tsconfig.json (updated)
package.json (updated)
```
`main.ts` (CLI script) will either be removed or kept as a dev-only script; its logic moves into `ResistorForm`/`ResultCard`.

## Phase 1 — Tooling & project restructuring
1. Update `package.json`: add `react`, `react-dom`, dev deps `vite`, `@vitejs/plugin-react`, `sass`, `@types/react`, `@types/react-dom`; add scripts `dev`, `build`, `preview`.
2. Add `vite.config.ts` with the React plugin.
3. Update `tsconfig.json` for a bundler-based React setup (`"jsx": "react-jsx"`, `"lib": ["ES2022", "DOM"]`, `"moduleResolution": "Bundler"`).
4. Create `index.html` + `src/main.tsx` entry point.
5. Move `network.ts` into `src/`; remove/retire root `main.ts` (CLI use case superseded by the UI).

## Phase 2 — React UI
1. `ResistorForm`: textarea for available values (parse comma **or** newline separated into `number[]`), text input for target value, "Calculate" button.
2. On submit, call `findResistorNetworkUnlimited(values, target, { relTolerance: 0.01 })` from `network.ts`.
3. `ResultCard`: renders `found/closest`, achieved value, resistor count, deviation %, and the `description` string (series/parallel expression).
4. Basic input validation (empty/invalid numbers, target <= 0) with inline error messaging.

## Phase 3 — Styling
1. Convert component styles to `.scss`, use a shared `_variables.scss` for colors/spacing.
2. Keep layout simple: form on top, result card below.

## Phase 4 — SST setup (latest v3 "Ion")
1. Add `sst.config.ts` defining the app and a single `sst.aws.StaticSite` resource pointing at the Vite build output (`dist`), with `build: { command: "npm run build", output: "dist" }`.
2. Define stages, ensure `int` stage config resolves (region, any env vars).
3. Add `sst` as a dev dependency; add `sst dev`/`sst deploy` scripts.

## Phase 5 — GitHub Actions workflow
1. `.github/workflows/deploy.yml`, triggered on push (branch TBD — see open question below).
2. Steps: checkout → `bahmutov/npm-install@v1` → `aws-actions/configure-aws-credentials@v6` → `npx sst deploy --stage int`.
3. Node version pinned via `actions/setup-node` (or rely on `npm-install` action's built-in Node handling — confirm during implementation).

## Phase 6 — Validation
1. `npm run build` succeeds locally.
2. `npm run dev` manual smoke test of the calculator UI.
3. `npx sst deploy --stage int` dry-run (or `sst diff`) if AWS credentials are available locally.

## Open questions before implementation
1. **AWS auth for the workflow** — use OIDC (`role-to-assume`) or static access-key secrets with `configure-aws-credentials@v6`? OIDC is more secure; default to it unless told otherwise, but a role ARN/secret name is needed.
2. **Trigger branch** — deploy `int` stage on push to `main`, or a dedicated `develop`/`int` branch?
3. **Package manager** — confirm npm (matches `bahmutov/npm-install@v1` and existing `package-lock.json`).
4. Keep the root CLI (`main.ts`) around for local testing, or fully remove it in favor of the web UI?
